# Trump

A 4-player mobile card game (2 teams of 2) built with **Godot 4.6** in **GDScript**, targeting **iOS** and **Android** with full crossplay. Supports single-player vs AI and online multiplayer (2–4 humans, empty seats filled by AI) backed by a dedicated headless server.

The canonical game rules, state machine, and product requirements live in [`CLAUDE.md`](./CLAUDE.md). This README covers how the project is built and how the code is organized — it is the orientation doc for a new engineer.

---

## Tech Stack

| Concern | Choice |
| --- | --- |
| Engine | Godot 4.6 (Mobile renderer, GL Compatibility) |
| Language | GDScript (typed) |
| Networking | Godot `MultiplayerAPI` over `ENetMultiplayerPeer` (UDP) |
| Server | Godot **headless** (`dedicated_server` feature) export, running as a Linux binary |
| Server hosting | Ubuntu VM on Proxmox homelab, fronted by Traefik (`server/traefik/trump-game.yml`) |
| iOS build | Xcode project under `Trump.xcodeproj/`, built from a Godot iOS export |
| Android build | Standard Godot Android export → APK |
| Persistence | `user://settings.cfg` (per-device settings); session stats in-memory only |

The wire protocol is intentionally minimal: every packet is a `Dictionary` of the form `{ "type": String, "data": Dictionary }`. All message names and error codes are defined in **one** file: `scripts/net/protocol.gd` — both client and server import it via `class_name Protocol`.

### What lives where — the load-bearing rules

These conventions are not enforced by the engine. Violating them will silently break the SP/MP split.

- **`scripts/server/` runs only on the dedicated server.** Client builds load these files but never instantiate the classes. Don't import server code from client code.
- **`scripts/net/protocol.gd` is the contract.** Any change to a message type or payload shape must be made here and rolled out to client *and* server in the same release. Never inline a magic string.
- **The UI never reads `RoundManager` directly.** It reads `GameState.game_source`, which is either `RoundManager` (single-player) or a `NetGameView` (multiplayer). Both expose the same surface. This is what makes the play screen mode-agnostic.
- **Autoloads are globals — keep them small.** `GameState` is the session, `NetworkState` is the connection, `Settings` is the disk. Game-loop logic belongs in `RoundManager` / `GameSession`, not in autoloads.
- **`bootstrap.gd` decides the instance's identity at startup** (server vs client) before any scene is loaded. See "Server vs Client mode" below.

## Setup — New Developer

### Prerequisites

| Tool | Version | Notes |
| --- | --- | --- |
| Godot | **4.6 stable** | Standard build for editing; install **export templates** matching the version (Editor → Manage Export Templates) |
| Xcode | 15+ | Required only for iOS device/TestFlight builds |
| Android SDK + JDK 17 | — | Required only for Android builds. Configure paths in Godot → Editor → Editor Settings → Export → Android |
| Git | any modern | — |

### First-time setup

```bash
git clone https://github.com/randyramsaywack/trump-mobile-card-game.git
cd trump-mobile-card-game
```

1. Open Godot 4.6 → **Import** → select `project.godot`.
2. Let Godot finish its first import pass (it will populate `.godot/`).
3. Press **F5** to run. The main scene is `scenes/main_menu.tscn`.

That's it for single-player development. The dev server / multiplayer flow below is only needed if you're working on networking.

### Running multiplayer locally

`bootstrap.gd` uses a **port-race** trick on desktop: the first instance that successfully binds the ENet port (`Protocol.SERVER_PORT`, default `9999`) becomes the server; subsequent instances detect the port is taken and start as clients pointing at `127.0.0.1`.

```
1st F5  →  server  (loads scenes/server_main.tscn)
2nd F5  →  client  (loads scenes/main_menu.tscn) → Multiplayer → Join Room
3rd F5  →  client
4th F5  →  client
```

To force server mode regardless of the race, pass `--server` on the command line, or run a headless export build (which sets the `dedicated_server` feature flag).

### Server address configuration

The default server address lives in `project.godot` under `[network]` (`server_address`, `server_port`). It is read once at startup by `Protocol`. To point at a different server without editing the file, override at the engine command line:

```bash
godot --main-pack ... --network-server-address=mydomain.example.com
```

(Or just edit `project.godot` for local dev.)

### Building the dedicated server

Use the **Linux/X11 — Headless Server** export preset in `export_presets.cfg`. This produces:

```
Trump.x86_64    # the binary
Trump.sh        # thin wrapper that execs Trump.x86_64 with passthrough args
Trump.pck       # game data
```

Both the binary and wrapper are gitignored — they are deploy artifacts. Ship them to the Ubuntu VM (e.g. `scp` to `/opt/trump/`) and run as a systemd service. Traefik routing for the public domain is configured in `server/traefik/trump-game.yml`.

### Building for iOS

1. Godot → Project → Export → **iOS** preset → **Export Project** to the repo root. This regenerates `Trump.xcodeproj/`.
2. Open `Trump.xcodeproj` in Xcode, set your signing team, archive, and upload to TestFlight.
3. Bump `Marketing Version` (`CFBundleShortVersionString`) and `Build` (`CFBundleVersion`) before each TestFlight submission.

`Trump.xcframework/` and `MoltenVK.xcframework/` are large binaries downloaded by the Godot iOS export and are gitignored. Re-exporting from Godot recreates them.

### Building for Android

Godot → Project → Export → **Android** preset → **Export Project** → produces `Trump.apk`. APKs are gitignored.

## Runtime — Server vs Client Mode

`autoloads/bootstrap.gd` runs once at startup and decides which scene to load. The decision tree (in priority order):

1. **`--server` flag OR `OS.has_feature("dedicated_server")`** → server. Binds the ENet port and loads `scenes/server_main.tscn`.
2. **Platform is iOS or Android** → always client. Loads `scenes/main_menu.tscn`. (Mobile must never accidentally become a host.)
3. **Desktop fallback** → race for the port. First instance wins server; the rest fall through to client.

The peer that wins the bind is stashed on `NetworkState.pending_server_peer` and consumed by `main_server.gd` in its `_ready()`. No global state changes hands after that.

## How a Frame Works

**Single-player:** `GameState.round_manager` (a `RoundManager`) drives the game loop directly. It owns the `Deck`, the four `Player` / `AIPlayer` seats, and the current `Trick`. UI subscribes to its signals (`round_started`, `card_played`, `trick_completed`, `round_ended`) and calls into it for human input.

**Multiplayer:** the same `RoundManager` lives on the **server**, owned by a `GameSession` inside a `Room`. The client runs a `NetGameView` that mirrors the server's state from broadcast messages and exposes the same signal surface to the UI. `GameState.game_source` is swapped between the two — the play screen (`game_table_ui.gd`) doesn't know which mode it's in.

This dual-source design is what keeps the UI from being forked. If you add a new game event, follow the existing pattern: emit a signal from `RoundManager`, mirror it on `NetGameView`, send a `Protocol.MSG_*` message to keep them in sync.

## Conventions & Gotchas

- **Equality over identity for cards.** Cards are passed across the wire as `{suit, rank}` dictionaries and re-instantiated on receive, so `Card == Card` only works via value comparison (suit + rank). Never use `==` on Card object identity in MP code paths — use `card.matches(other)` or compare the suit/rank fields.
- **Portrait-only.** Viewport is `390 × 844` (iPhone 13 logical). Don't add landscape layouts.
- **Shared `protocol.gd` is a versioning boundary.** Bumping a message shape requires synchronized server + client deploys.
- **Screenshots verify UI work.** See the "Self-Verification via Screenshots" section in `CLAUDE.md` — visually confirm changes before declaring them done.

## Where to Look First

- New to the rules? → `CLAUDE.md` (Game Rules section).
- Adding a UI screen? → `scripts/ui/` + `scenes/ui/`. Wire it into `main_menu_ui.gd` or as an overlay.
- Adding a networked event? → `scripts/net/protocol.gd` first, then server (`scripts/server/game_session.gd`), then client (`scripts/net/net_game_view.gd`).
- Tweaking AI? → `scripts/ai_player.gd`.
- Changing the game-loop rules? → `scripts/round_manager.gd` (and matching server logic in `scripts/server/game_session.gd`).
